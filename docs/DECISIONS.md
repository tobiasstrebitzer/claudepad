## Resolved

| # | Decision | Rationale |

## Consolidation reconciliations (2026-06-20 review pass)

Cross-PRD inconsistencies found while reviewing the nine drafts and how they were resolved:

1. **Wire contract (PRD-05 ↔ PRD-07).** PRD-05 had drafted a JSON-wrapped `POST /api/blobs` body, a `meta.expiry` enum incl. `never`, fetch at `GET /s/:id`, and `HEAD ?peek=1`. PRD-07 (the API owner) specifies an `application/octet-stream` body with `Cp-*` headers, `POST /v1/blobs`, `GET /v1/blobs/:id`, `/meta` peek, and `/s/:id` as the SPA shell. → **PRD-05 §7.4/FR-20/§4.1/§4.2/§6.6/Q-3 edited to match PRD-07.** (D-18.)
2. **Default TTL.** PRD-05 said 7d; PRD-07 said 30d. → **Standardized on 7d** (`DEFAULT_TTL=604800`s); PRD-07 FR-2 updated. (D-14.)
3. **`never` expiry.** PRD-05 offered it; PRD-07 forbade it. → **Removed from hosted v1**, self-host config-gated via `MAX_TTL`. (D-16.)
4. **`deleteToken`.** PRD-07 introduced it; PRD-05's `CreateResponse` didn't have it. → **PRD-05 `CreateResponse` updated** to include `deleteToken`/`url`. (D-17.)
5. **Type-spec authority.** PRD-02 finalized a richer schema than `_context.md` §6's sketch. → **`_context.md` §6 now defers to PRD-02** as authoritative.

## Serverless-v1 pivot (2026-06-20) - supersedes where noted

After brainstorming identity-based sharing and **building a working proof of concept (`poc/`)**, v1 is re-scoped to be **entirely client-side**. The recipient-encrypted-blob model needs no server, so we trade server-dependent features for true trustlessness. These decisions take precedence over the earlier "Hosted Sharing (P2)" framing.

| # | Decision | Rationale |
|---|----------|-----------|
| D-20 | **v1 is entirely client-side (serverless).** `claudepad.io` ships as a static site; no backend, no accounts, no hosted URLs. | The trustless blob model needs no server. Trustlessness + zero-ops self-host > server-only features. (User decision, 2026-06-20.) |
| D-21 | **Primary sharing model = trustless recipient-encrypted blobs.** Sender encrypts to the recipient's public key; the blob *is* the message, carried via clipboard/file - "drop it anywhere, only the invited person can read it." | Proven in `poc/` (21/21 crypto checks). No host can read or gatekeep. |
| D-22 | **Identity = client-minted ECDH P-256 keypair** + self-claimed display name; exportable secret; no server account. | P-256 has zero-dependency WebCrypto support (X25519 needs a lib). Pseudonymous, Nostr-like. X25519 documented as the alt. |
| D-23 | **Per-recipient tiering via key wrapping.** Independent `K_body`/`K_secret` (D-13) are wrapped to each recipient's pubkey; `wrappedSecret` included only for high-priv recipients. | Replaces the two-fragment-link tiering for v1; cleaner (one blob, per-person tier) and falls out of the recipient model. |
| D-24 | **Ephemeral sealed-box per share** (ephemeral ECDH keypair → HKDF-SHA256 → AES-256-GCM wrapping key). Sender identity embedded for display. | Unlinkable recipient sets across shares + forward secrecy on the wrap. |
| D-25 | **Key fingerprints** (SHA-256 over the raw public key → 6 emoji + 8-hex code) for out-of-band trust. | The name is self-claimed; a short human-comparable fingerprint is the Signal-style verification (mitigates MITM-on-pubkey). |
| D-26 | **Optional device protection via WebAuthn PRF (pattern A default).** Passkey/security-key PRF output derives a KEK that wraps the stored private key - client-side only, no server. PRF evaluated **at registration** to avoid a double prompt. | Hardware-gated identity at rest, no backend. Pattern B (PRF-as-seed → multi-device via synced passkeys) documented; needs `@noble/curves` for deterministic keygen. Exported secret stays the recovery fallback. |
| D-27 | **Crypto core stays zero-dependency WebCrypto** for v1 (AES-256-GCM, ECDH P-256, HKDF-SHA256, SHA-256). | Auditability + single-static-bundle goal. `@noble/curves` enters only if pattern-B identity ships. |
| D-28 | **`poc/` is the reference implementation** of the crypto/identity/sharing core; `poc/verify.mjs` (21 checks) is the conformance anchor for the production code. | De-risks the spec; PRDs describe what the PoC already proves. |
| D-29 | **Deferred to vNext (consciously giving up):** server blob store (PRD-07), hosted `/s/<id>` short URLs, fragment-link/`#key` mode, expiry/TTL, burn-after-read, revocation, inbox/discovery, availability/pinning. | These all require a server or mutable/coordinated storage. Re-introduce only as an optional convenience layer later. |

**Status changes from the pivot:** **D-14, D-16, D-17, D-18** (server lifecycle, wire contract, delete tokens) are **moot for v1** and retained only for the vNext backend. **PRD-07** is **Deferred (vNext)**. **PRD-05** is re-scoped from "link-mode crypto + backend wire contract" to "client crypto core & recipient wrapping." Fragment-link grammar moves to vNext.

## Pluggable store as an open spec (2026-06-20)

The backend, when it returns, is an **optional, opt-in addon defined by an open contract** - not a proprietary service. Model: Bitwarden (self-host the server, point the client at any URL) / Tailscale↔Headscale (open protocol, reference control server).

| # | Decision | Rationale |
|---|----------|-----------|
| D-30 | **The store is a SPEC, not a service.** "PRD-07" is re-scoped to the **Store Provider Specification** (an open HTTP contract for a zero-knowledge blob store) plus a **reference implementation** hosted at `claudepad.io/store`. Any org can implement the spec; trusting users can point at `claudepad.io/store`. | Avoids vendor lock-in; the value is the open contract. `claudepad.io/store` is *one* implementation, free & open-source. |
| D-31 | **The store is a pluggable, opt-in addon.** Default = **no store** (pure client). A store is configured by URL (self-host, org, or `claudepad.io/store`); the bundle calls it only through the `StoreProvider` interface. | Keeps the root bundle pure; users choose if/whom to trust with hosting their (already-encrypted) blobs. |
| D-32 | **v1 ships the clean seam, not the store.** The static bundle defines the `StoreProvider` interface and a default **`NoStoreProvider`** (everything stays client-side); **no provider implementation and no `claudepad.io/store` URL or store-specific code** ship in v1. | v1 has zero store dependency and can launch without it, while the integration point exists so the addon drops in later without re-architecture. |
| D-33 | **No `claudepad.io/store`-specific code or mentions in the v1 client.** The store URL is runtime/build config with default empty; the core is store-agnostic. | The store must remain a swappable spec implementation, never hardwired. |

### Registry spec - store + identity directory (2026-06-22)

The store seam (D-30…D-33) is **extended** into a **registry**: still vNext, still an open spec, still null-by-default in the v1 client, but now spanning three axes - availability (host the blob), authenticity (a public-key directory), and confidentiality (default ZK, opt-in trusted). Canonical doc: `REGISTRY-SPEC.md` (absorbs `STORE-PROVIDER-SPEC.md`).

| # | Decision | Rationale |
|---|----------|-----------|
| D-74 | **A "registry" = the store seam extended along two new axes (an identity *directory* and a *publish* target), kept as an open spec - not a service - and still vNext.** v1 ships only the `RegistryProvider`/`RegistryManifest` *types* alongside the existing `StoreProvider`; **no implementation, no `claudepad.io/registry` URL, no directory/trusted-mode code** in the v1 client (extends D-32/D-33). Default = no registry; every flow still works fully offline. | A registry is the natural team/convenience addon (short links, share-by-name, org archive) and must drop in without re-architecture. Keeping it spec-not-service and null-by-default preserves the trustless launch (D-20…D-29) and the "no store code in the client" guarantee. |
| D-75 | **Confidentiality is a per-registry *mode*: zero-knowledge by default, trusted as an explicit opt-in.** ZK = client encrypts-to-recipient (`createBlob`/`createMultiBlob`, using public cards from the directory) and the registry stores **opaque ciphertext** it cannot read - identical crypto to carry-the-blob sharing. Trusted = the client uploads a **readable** session over TLS (registry encrypts at rest, can index/search/render) for orgs that want server-side access; the client **must label this ("this registry can read this session") and require explicit consent** before the first trusted publish. A registry declares its `modes` in the manifest; the client never switches silently. | Preserves claudepad's whole pitch (the registry sees ciphertext) while giving teams the readable-archive escape hatch they will ask for - with eyes open. Honesty over polish: the trust shift is shown, not buried. An optional ZK *inbox* index (`indexFor`) reveals the social graph (not content) and is therefore off by default. |
| D-76 | **Authenticity = the directory declares a per-identity *assurance level* (`self` / `domain` / `sso`); the client always surfaces assurance *and* fingerprint.** `self` = the registry vouches for nothing (a cache) -> the v1 out-of-band fingerprint confirm is **unchanged**. `domain`/`sso` = the registry verified the key<->person binding -> the client may skip the manual fingerprint dance but **must** show "verified by `<registry>` via ..." + the fingerprint; trust then reduces to "I trust this registry." This consciously relaxes `TRUSTLESS-MODEL.md` §8 ("not a key directory/PKI") **for the opt-in addon only**, honestly framed. | "Share with Dana" instead of a key-exchange ritual is the friction win for verified teams, but it must never pretend an unverified key is verified. Surfacing the assurance level + fingerprint keeps the trust explicit and lets a skeptic fall back to `self` semantics. Only public cards live in the directory; private keys never touch a registry. |
| D-77 | **TLS is mandatory for every registry interaction; the social-graph metadata trade-off is documented, not hidden.** The client rejects any non-`https://` registry `baseUrl` (only `http://localhost` allowed for dev); a manifest must declare `tls: 'required'`. Configuring a registry adds exactly one `connect-src` origin - the bundle still ships no third-party scripts/CDNs. The opt-in inbox index leaks who-shares-with-whom to the registry (content stays encrypted) and is off by default. | A registry is a network surface; HTTPS-only is the floor. In ZK mode TLS termination reveals nothing (payload is already ciphertext); in trusted mode TLS protects the plaintext upload. Documenting the inbox social-graph leak upholds the honesty principle (`TRUSTLESS-MODEL.md` §7). |
| D-80 | **The registry is wired into the shipping client as an opt-in, null-by-default integration (2026-06-22) - consciously evolving D-32's "ship the seam, not the store" now that the addon is being built.** A `RegistryProvider`/`useRegistry` context holds a persisted registry URL (empty default, https-only via the SDK guard) and connects through `@claudepad/registry-client`; a sidebar-footer `RegistryControl` configures it. Surfaces: short-link **upload** + **receive-by-id** (ZK, opaque blobs), **share-by-name** (directory lookup in the recipient step - verified `domain`/`sso` entries add directly, `self` routes through the existing out-of-band fingerprint confirm), **publish-your-identity** to a directory, opt-in **inbox** ("shared with me") in the receive dialog, and consent-gated **trusted-mode publish** of the redacted body. Invariants preserved: **no hardwired `claudepad.io` URL**, every flow works with no registry, and the `check-no-external-origins` gate stays green (the OpenAPI doc with its `example.com` URLs is kept off the client graph via the `@claudepad/registry-spec/openapi` subpath, D-79). | The user opted into full integration; with the external-review gate dropped (D-78) there is no launch-purity blocker. The generic client (no hardwired origin, empty default) honors D-33's letter ("store-agnostic, URL is plain config") while going past D-32's seam-only stance - the honest framing is that v1 now *contains* an optional registry client, not that it depends on one. Opt-in inbox indexing (`indexFor`) and trusted publish each carry an explicit in-UI trust notice (D-75/D-77). |
| D-79 | **The registry contract + reference implementation + client SDK are built as three new packages, fully outside the v1 client bundle (2026-06-22).** `@claudepad/registry-spec` (zero-dep contract: provider interfaces, wire DTOs, endpoint paths, typed errors, the HTTPS-only guard, a tolerant manifest parser, the OpenAPI 3.1 doc), `@claudepad/registry` (Cloudflare Worker over a storage-agnostic handler + `InMemoryBackend`; R2 blobs/sessions, KV directory/inbox; reference-grade bearer auth; derives directory fingerprints authoritatively via `@claudepad/shared` and issues only `self` assurance - no verifier), and `@claudepad/registry-client` (a `fetch`-based `RegistryClient` that works against any conformant registry, no hardwired URL, with a conformance suite run against the reference handler). The shipping client (`@claudepad/client`) is **untouched** - no registry import, no URL - so D-33/D-74 and the trustless launch hold. Inbox `indexFor` keys are treated as **opaque** by the registry (pubHash semantics stay client-side, OQ-R2). | Builds the optional addon as a clean drop-in without contaminating the pure client. A storage-agnostic handler makes the contract runnable in-process (in-memory backend), so the SDK conformance suite and the Worker share one code path and can't drift. Authoritative fingerprint derivation + honest `self`-only assurance match D-76 (the reference impl can't verify domain/SSO, so it doesn't claim to). |

### Viewer themes decisions (aesthetic palette axis, PRD-12, 2026-06-22)

| # | Decision | Why |
